This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project — EnergyAnomaly (Kaggle LEAD)
[`energy-anomaly-detection`](https://www.kaggle.com/competitions/energy-anomaly-detection):
binary classification of whether each **hourly electricity meter reading** is anomalous (`1`) or
normal (`0`). Full-year 2016 hourly data. Train: 200 buildings (~1.75M rows). Test: 206 buildings.
**Metric: AUC-ROC** — submissions are probabilities (`row_id,anomaly`), not hard labels, so there is
no threshold to tune.

**The single most important rule: CV must be `GroupKFold` by `building_id`.** Train and test
buildings are **disjoint** (0 overlap), so generalization is *across buildings*, not across time. A
random/stratified KFold leaks per-building behaviour and massively overstates OOF.

## Environment & commands
Uses **uv** (not pip/`requirements.txt` like the sibling Kaggle projects in this monorepo). Deps are
pinned in `pyproject.toml` / `uv.lock`. **Run everything from this project dir (`EnergyAnomaly/`)**,
and prefix Python with `uv run` so the project venv is used:

```bash
uv sync                                              # create/refresh the venv
uv run pytest                                        # all tests (once tests exist)
uv run pytest tests/test_features.py::test_name      # a single test
uv run python -m src.train --model lgbm              # (planned) CV run -> OOF + bagged test
```

**Kaggle auth uses a newer `KGAT_` token at `~/.kaggle/access_token` (no `kaggle.json`).** The CLI
reads it from an env var:

```bash
export KAGGLE_API_TOKEN="$(cat ~/.kaggle/access_token)"
uv run kaggle competitions download -c energy-anomaly-detection -p data/raw
unzip -o data/raw/energy-anomaly-detection.zip -d data/raw
```

## Data
Each split ships in two flavours under `data/raw/` (git-ignored):
- `train.csv` / `test.csv` — minimal: `building_id, timestamp, meter_reading (+anomaly on train)`.
- `train_features.csv` / `test_features.csv` — same rows **plus ~50 pre-engineered columns**:
  building metadata, weather, weather lag-aggregates, calendar (raw + cyclical `*_x/_y`), and
  **`gte_*` group-target-encoding** columns. Build off these rather than re-engineering from scratch.

**Data facts that drive the model** (full analysis in `reports/EDA_FINDINGS.md`):
- **`meter_reading == 1.0` → anomaly in 99.97%** of cases — a stuck/flatlined meter. It is ~46% of
  all anomalies. Encode it as an `is_reading_one` flag **plus flatline run-length features**, not a
  hard `==1.0` override (run-length generalizes to unseen test buildings; the other ~54% of anomalies
  are normal-magnitude contextual cases the model must still learn).
- `meter_reading` is missing in ~6.2% of rows and **missing readings are never anomalies** → predict
  low there.
- `gte_*` columns are **target-derived and leakage-prone** (likely fit on full train). Audit how they
  were built; recompute inside folds or drop them to keep OOF honest.
- Weather columns look complete only because missing was **sentinel-imputed** (e.g.
  `cloud_coverage == 255` for ~46% of rows). Restore sentinels to `NaN` + add missing-flags, and
  re-encode `wind_direction` (circular) as `sin/cos`. Weather is kept (trees use it via interactions),
  not excluded.

## Architecture (prediction-file pipeline — planned, mirrors sibling `Diabetes/`)
`src/` is not built yet. The intended design communicates through **saved prediction arrays**
(`{model}_oof.npy` + fold-averaged `{model}_test.npy` in `experiments/artifacts/`) so training and
submission are decoupled and any subset of models can be blended after the fact:
- `src/data.py` — schema constants + loaders (raise on schema mismatch); cache merged frame to
  `data/processed/`.
- `src/features.py` — leakage-safe feature engineering (flatline/run-length, missing flags, sin/cos
  wind, per-building lag/value-change features computed within building & time order).
- `src/cv.py` — `GroupKFold(building_id)`.
- `src/train.py` — grouped CV loop scoring OOF probabilities with `roc_auc_score`; saves OOF + bagged
  test arrays. Models: LightGBM / XGBoost / CatBoost / HistGradientBoosting.
- `src/blend.py` — equal-weight **rank-average** of base learners (prefer over OOF-optimized stacking).
- `src/submit.py` — build + validate submission (shape/ids/range vs `sample_submission.csv`), optional
  Kaggle upload.

Optional GPU deep-learning track (`src/dl/`, PyTorch) produces OOF/test scores in the same `.npy`
format to add as an ensemble member. The GBDT track is CPU-only.

Git-ignored: `data/`, `models/`, `outputs/`, `experiments/artifacts/`, `reports/figures/`,
`.venv/`, and the kaggle token.

## EDA
`notebooks/01_eda.ipynb` is **generated by a build script** (a Python file using `nbformat`), then

the generator and re-run, rather than hand-editing notebook JSON. Figures are saved to
`reports/figures/`; written conclusions live in `reports/EDA_FINDINGS.md`.

## Conventions & hard-won lessons
- AUC is rank-based: don't reflexively reweight classes; validate `scale_pos_weight`/downsampling on
  OOF before adopting (it often doesn't help AUC).
- Expect a CV↔LB gap and do not chase OOF; an **equal-weight diverse blend** tends to generalize
  better than OOF-optimized weighting (sibling-project lesson).
- **Show CV/OOF metrics and get explicit approval before any Kaggle submission.** Log every run in
  `experiments/README.md`. Commit incrementally after each meaningful step; stage only this project's
  files (the monorepo contains unrelated sibling projects).
- The living plan for this project is at
  `~/.claude/plans/think-through-the-steps-dynamic-star.md`.
